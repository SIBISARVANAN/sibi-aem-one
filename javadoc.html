Project Overview

sibi-aem-one is an Adobe Experience Manager (AEM) Maven project that demonstrates a wide range of AEM development patterns including Sling Models, OSGi Services, Schedulers, Sling Jobs, Event Listeners, Sling Filters, Sling Servlets, QueryBuilder-based Search, and Workflow Process Steps.

ResponseModificationFilter

filters

Demonstrates response body modification using the BufferedHttpResponseWrapper. Before calling chain.doFilter(), wraps the response to intercept its output. After the servlet/component renders, reads the buffered HTML and injects a tracking script tag before </body>. Updates the Content-Length header to reflect the modified size before writing to the real response.

Performance note: Response wrapping holds the entire page in memory. Only use on low-traffic, non-cacheable paths. Never use on high-traffic paths without profiling first.

doFilter() Flow

StepAction
1Wraps the response in BufferedHttpResponseWrapper to intercept output.
2Calls chain.doFilter(request, wrappedResponse) — servlet writes to the buffer.
3Reads the buffered HTML string via wrappedResponse.getBufferedContent().
4Replaces </body> with a tracking script tag + </body>.
5Sets correct Content-Length and writes modified HTML to the real response.

An HttpServletResponseWrapper that captures all output written to the response into an in-memory ByteArrayOutputStream. Both the text-based getWriter() path and the binary getOutputStream() path are intercepted. Used by ResponseModificationFilter to buffer the rendered HTML before modifying it.

AuthorUpdateClusterScheduler

schedulers

Cluster-safe version of AuthorUpdateScheduler that uses JVM-level and JCR-level locking to ensure only one AEM instance in a shared-repository cluster runs the job at a time. Combines a ReentrantLock (JVM guard) with JCR LockManager (cluster guard). Useful in MongoMK/TarMK-shared setups where multiple author instances share a single Oak repository.

Cluster safety: Two levels of locking — ReentrantLock prevents two threads on the same JVM, and JCR LockManager prevents two JVMs in the same cluster from running simultaneously.

Locking Strategy

Lock TypeScopeImplementation
JVM LockSingle AEM instanceReentrantLock.tryLock() — non-blocking; skips execution if already locked.
JCR LockEntire clusterLockManager.lock(LOCK_PATH, false, true, 300, null) — creates a session-scoped, 5-minute lock on a dedicated JCR node.

Key Methods

MethodDescription
run()First attempts JVM lock. If acquired, attempts JCR cluster lock. If both succeed, executes business logic. Both locks are released in finally blocks.
getClusterLock(ResourceResolver)Creates the lock node path if it doesn't exist, checks if already locked, then acquires a JCR lock with a 300-second timeout.
releaseClusterLock(ResourceResolver)Checks if the lock still exists and unlocks it via LockManager.unlock().
getResourceResolver()Opens a service-user-based resource resolver using subservice "scheduler".

ClusterAwareListener

listeners

Implements both ResourceChangeListener and ExternalResourceChangeListener, making it aware of changes from both the local node and remote cluster nodes. Uses change.isExternal() to route events to different job topics: internal changes get a local job and external (cross-cluster) changes get an external job, preventing duplicate job execution.

Key pattern: Always check isExternal() before firing a job. Without this check, a 2-node author cluster would fire 2 identical jobs for every change.

Change Filtering Chain in onChange()

Filter StepLogic
1. Noise filterSkips changes whose path contains jcr:content/ (sub-property noise).
2. Page-level filterKeeps only jcr:content nodes or non-JCR paths.
3. Type filterIgnores REMOVED events (handles only ADDED and CHANGED).
4. Local/external branchRoutes to .job.internal or .job.external topic based on isExternal().

OSGi Service (v1)

GoogleRecaptchaConfigServiceImpl (v1)

services.impl.v1

Container-managed lifecycle pattern. The service instances register themselves into a static registry via a companion Registry inner class that uses @Reference(cardinality=MULTIPLE, policy=DYNAMIC) bind/unbind methods. The OSGi container manages when to call bind/unbind — the developer only reacts to those events. This is the container-managed variant of the multi-instance registry.

Design: The inner Registry class is a separate @Component that holds the static ConcurrentHashMap. Whenever a new GoogleRecaptchaConfigService OSGi service appears or disappears, the container calls bind() / unbind() on Registry, which updates the map. Consumers call Registry.getGoogleRecaptchaConfigService(siteName) to look up a site.

getGoogleRecaptchaConfigService(siteName)Static lookup — returns the service for the named site.
getRegistry()Returns an unmodifiable view of the full registry map.
OSGi Service (v2)

GoogleRecaptchaConfigServiceImpl (v2)

services.impl.v2

Application-managed lifecycle pattern. Each factory instance directly manages its own registration: @Activate calls REGISTRY.put(siteName, this) and @Deactivate calls REGISTRY.remove(siteName). Simpler structure than v1 — no inner class, no dynamic references — but the developer owns thread safety and lifecycle guarantees. Both patterns produce the same runtime result: a map of site-specific service instances.

v1 vs v2: v1 (container-managed via bind/unbind) is safer for dynamic hot-swap and multi-bundle scenarios. v2 (application-managed via @Activate/@Deactivate) is simpler to write and sufficient for most use cases where all factory instances are in the same bundle.
